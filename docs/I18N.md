# Internationalization Guide

900Invoice supports 6 languages and is designed to make adding new languages straightforward. This guide explains the i18n system, how to add a new language, and how to handle special cases like RTL scripts and locale-specific number formatting.

---

## Overview

The i18n system uses JSON translation files located in `src/i18n/`. The frontend loads the appropriate file based on the user's language setting (stored in the `settings` table) and provides a `t()` function for components to use.

**Current languages:**

| Code | Language | Direction | Status |
|------|----------|-----------|--------|
| `en` | English | LTR | Complete (reference) |
| `fr` | French | LTR | Complete |
| `es` | Spanish | LTR | Complete |
| `ar` | Arabic | RTL | Complete |
| `sw` | Swahili | LTR | Complete |
| `hi` | Hindi | LTR | Complete |

## File Structure

```
src/
├── i18n/
│   ├── en.json       # English — the reference/base language
│   ├── fr.json       # French
│   ├── es.json       # Spanish
│   ├── ar.json       # Arabic (RTL)
│   ├── sw.json       # Swahili
│   └── hi.json       # Hindi
├── stores/
│   └── i18nStore.ts  # Translation loading, t(), locale registry, RTL dir/lang handling
└── utils/
    └── locale.ts     # Date and number locale mapping
```

The `en.json` file is the canonical reference. All other languages must contain every key that exists in `en.json`. Missing keys fall back to the English value.

## Key Naming Conventions

Keys are dot-namespaced by feature and context:

```json
{
  "app": {
    "name": "900Invoice"
  },
  "nav": {
    "dashboard": "Dashboard",
    "invoices": "Invoices"
  },
  "invoices": {
    "new": "New Invoice",
    "number": "Invoice #",
    "filterDraft": "Draft",
    "filterFinalized": "Finalized",
    "filterSent": "Sent",
    "filterPaid": "Paid",
    "filterVoid": "Void"
  },
  "clients": {
    "name": "Client Name",
    "email": "Email"
  },
  "taxes": {
    "vat": "VAT",
    "gst": "GST",
    "withholding": "Withholding Tax"
  },
  "common": {
    "save": "Save",
    "cancel": "Cancel",
    "delete": "Delete",
    "edit": "Edit",
    "loading": "Loading...",
    "error": "Something went wrong"
  }
}
```

### Naming rules:
- Use nested objects by feature area.
- Components read nested values with dot paths such as `t('invoices.new')`.
- Group by feature: `invoices.*`, `clients.*`, `products.*`, `payments.*`, `reports.*`, `settings.*`, `common.*`.
- Use the existing camelCase style for multi-word keys such as `filterDraft` and `businessProfile`.

## Placeholder Interpolation Syntax

Use `{variable_name}` (single curly braces) for dynamic values within translation strings:

```json
{
  "invoices": {
    "itemsCount": "This invoice has {count} line items"
  },
  "clients": {
    "welcome": "Welcome back, {name}"
  },
  "payments": {
    "recorded": "Payment of {amount} recorded on {date}"
  }
}
```

In Svelte components:
```typescript
import { t } from '../stores/i18nStore';

// Single variable
t('invoices.itemsCount', { count: lineItems.length })
// → "This invoice has 3 line items"

// Multiple variables
t('payments.recorded', { amount: 'KES 1,500.00', date: '2026-05-11' })
// → "Payment of KES 1,500.00 recorded on 2026-05-11"
```

**Do not** use string concatenation for user-visible text:
```typescript
// WRONG — breaks translations and word order
const msg = t('common.loading') + ' ' + clientName + '...';

// CORRECT
// Add a key: "client.loading": "Loading {name}..."
const msg = t('clients.loading', { name: clientName });
```

Word order differs between languages. French often puts adjectives after nouns. Arabic and Hebrew read right-to-left. Languages like Japanese use different sentence structure entirely. Always use placeholder interpolation, never concatenation.

## How to Add a New Language

Follow these steps to add a new language to 900Invoice. This is one of the most impactful contributions you can make.

### Step 1: Copy the English base file

```bash
cp src/i18n/en.json src/i18n/XX.json
# Replace XX with the ISO 639-1 language code
# Examples: pt (Portuguese), de (German), zh (Chinese), ur (Urdu), am (Amharic)
```

### Step 2: Translate all values

Open `src/i18n/XX.json` and translate every value. Rules:
- **Do not** change the keys (left side of `:`).
- **Do not** remove any keys.
- **Do** preserve `{variable}` placeholders exactly as written — only translate the surrounding text.
- **Do not** translate proper nouns like "900Invoice", "USD", "KES", "VAT", or country names that have no conventional translation.

Example (English → Portuguese):
```json
{
  "app": {
    "name": "900Invoice"
  },
  "nav": {
    "dashboard": "Painel"
  },
  "invoices": {
    "new": "Nova Fatura",
    "itemsCount": "Esta fatura tem {count} itens"
  },
  "common": {
    "save": "Salvar"
  }
}
```

### Step 3: Register the language

In `src/stores/i18nStore.ts`, add a `loadTranslations()` case for your JSON file:

```typescript
case 'pt':
  data = (await import('../i18n/pt.json')).default as Record<string, unknown>;
  break;
```

Then add your language to the `SUPPORTED_LOCALES` array in the same file:

```typescript
export const SUPPORTED_LOCALES = [
  { code: 'en', name: 'English', nativeName: 'English' },
  { code: 'fr', name: 'French', nativeName: 'Français' },
  { code: 'es', name: 'Spanish', nativeName: 'Español' },
  { code: 'ar', name: 'Arabic', nativeName: 'العربية' },
  { code: 'sw', name: 'Swahili', nativeName: 'Kiswahili' },
  { code: 'hi', name: 'Hindi', nativeName: 'हिंदी' },
  { code: 'pt', name: 'Portuguese', nativeName: 'Português' },
];
```

Use `nativeName` to write the language name in its own script. This is what appears in the language picker so users can identify their language without reading English.

### Step 4: Configure locale for date and number formatting

In `src/utils/locale.ts`, add your locale's formatting preferences:

```typescript
export const LOCALE_CONFIG: Record<string, LocaleConfig> = {
  // ...existing entries...
  pt: {
    code: 'pt',
    dateLocale: 'pt-BR',       // or 'pt-PT' for European Portuguese
    numberLocale: 'pt-BR',
  },
};
```

If your locale is not listed, the formatter uses the browser/app locale directly.

### Step 5: For RTL languages

If your language is written right-to-left (Arabic, Hebrew, Urdu, Persian, etc.):

1. Update the RTL branch in `setLocale()` inside `src/stores/i18nStore.ts` to include your locale code.
2. The i18n store applies `dir="rtl"` to the `<html>` element when an RTL language is active.
3. Test all screens in RTL mode — see [RTL Testing](#rtl-testing) below.

### Step 6: Test your translation

1. Run `cargo tauri dev`
2. Go to Settings → Language → select your language
3. Navigate through every screen and check:
   - All text appears in the correct language
   - No raw keys are visible (e.g., `invoices.filterDraft` should show as your translation)
   - `{variable}` placeholders are correctly substituted with values
   - Dates and numbers are formatted correctly for your locale
4. Create a test invoice, generate a PDF, and verify the preview uses your translations while native PDF export uses the saved locale, paper size, and date-format settings

### Step 7: Update documentation

Add your language to the table in this file (docs/I18N.md) and in the README.md Features section.

### Step 8: Open a Pull Request

Open a PR with the title: `feat(i18n): add Portuguese translation`

Include in the PR description:
- The language and code you added
- Whether it is LTR or RTL
- How you tested it (ideally with screenshots)
- Your name as you would like to be credited (optional)

## RTL Considerations

### How RTL works in 900Invoice

When the user selects an RTL language (currently Arabic), `setLocale()` in the i18n store sets `dir="rtl"` on the `<html>` element. Modern browsers automatically mirror the layout for RTL when `dir="rtl"` is set:

- Flex rows reverse direction
- Text alignment defaults to right
- Padding and margin start/end properties flip

### CSS logical properties

All CSS in 900Invoice should use logical properties instead of physical `left`/`right`:

```css
/* Use logical properties — these flip automatically with dir="rtl" */
padding-inline-start: 16px;   /* = padding-left in LTR, padding-right in RTL */
padding-inline-end: 16px;
margin-inline-start: 8px;
border-inline-start: 1px solid var(--border);

/* Avoid physical properties unless direction-independent */
padding-left: 16px;   /* Does NOT flip — avoid */
```

### RTL Testing

When adding or modifying UI components, test them in RTL mode:

1. Go to Settings → Language → Arabic
2. Check that:
   - Text is right-aligned
   - Icons and arrows point in the correct direction (back arrows should point right in RTL)
   - Form inputs are right-aligned
   - Tables render correctly
   - Dropdown menus open from the correct side
   - Modals and sidebars align correctly
   - The navigation sidebar appears on the right

### RTL in PDFs

The native PDF export currently uses built-in PDF fonts and a left-to-right text layout for maximum portability. The WebView preview and application UI support RTL; full RTL PDF shaping is tracked as a future rendering enhancement.

## Currency and Date Formatting

### Currency Formatting

Currency amounts are formatted in the frontend using `Intl.NumberFormat` with the active app locale:

```typescript
// src/utils/currency.ts
export function formatCurrency(
  minorUnits: number,
  currencyCode: string,
  position: 'before' | 'after' = 'before',
  locale?: string
): string {
  const config = getCurrencyConfig(currencyCode);
  const decimals = config.decimals;
  const amount = minorUnits / Math.pow(10, decimals);
  const formatted = new Intl.NumberFormat(resolveNumberLocale(locale), {
    minimumFractionDigits: decimals,
    maximumFractionDigits: decimals,
  }).format(amount);
  return position === 'after'
    ? `${formatted} ${config.symbol}`
    : `${config.symbol} ${formatted}`;
}
```

This produces locale-correct output:
- `en`: `KSh 10,000.00`
- `fr`: `KSh 10 000,00`
- `ar`: `KSh ١٠٬٠٠٠٫٠٠`

Native PDF export reads the persisted locale setting before rendering. It uses locale-specific decimal and thousands separators, while using ISO currency codes in the PDF text for compatibility with built-in PDF fonts.

### Date Formatting

Configured numeric date formats still use the selected date pattern. Month-name formats and relative timestamps use `Intl.DateTimeFormat`/`Intl.RelativeTimeFormat` with the active app locale:

```typescript
export function formatDate(isoDate: string, format = 'MMM D, YYYY', locale?: string): string {
  return new Intl.DateTimeFormat(resolveDateLocale(locale), {
    year: 'numeric',
    month: 'short',
    day: 'numeric',
  }).format(new Date(isoDate));
}
// en: "Mar 5, 2026"
// fr: "5 mars 2026"
// ar: "٥ مارس ٢٠٢٦"
// sw: "5 Mac 2026"
```
